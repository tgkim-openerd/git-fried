## 2. v1.1 — EV 코드 서명 + Tauri auto-updater (~0.5~1개월)

### 진입 조건
- v1.0 GitHub releases 첫 배포 완료
- GitHub repo (`tgkim/git-fried`) public + CI 파이프라인 작동
- 사용자 본인 환경에서 30+ 일 dogfood 안정 확인

### 결정 항목 (5)
1. **OV → EV 시기**: 즉시 EV 도입 vs 6개월 OV 후 EV
   - 권장: **EV 직행** ($400/yr) — SmartScreen 평판 누적 시간 절약. R7 정합
2. **인증서 vendor**:
   - Sectigo EV ~$400/yr
   - DigiCert EV ~$500/yr + reputation 빠름
   - SSL.com EV ~$300/yr (예산)
   - 권장: **SSL.com 또는 Sectigo** (가성비)
3. **HSM**:
   - Hardware token (Yubikey FIPS 140-2) ~$200 1회
   - eToken 5110 같은 vendor token ~$150
   - 권장: **eToken 5110** — vendor 호환성 검증됨
4. **Updater endpoint**:
   - GitHub releases 의 `latest.json` 직접 호스트 (무료)
   - 자체 endpoint (`updater.git-fried.dev`) — DNS + S3 ~$5/mo
   - 권장: **GitHub releases 직접** (단순, 무료, 신뢰)
5. **Updater 정책**:
   - check-on-startup vs daily background check
   - 자동 다운로드 vs 사용자 confirm
   - 권장: **startup check + 사용자 confirm** (R7 신뢰 누적, no surprise)

### 작업 분해
| # | 작업 | 시간 |
| - | --- | --- |
| 1 | EV 인증서 구매 + HSM 도착 + 검증 | 1~2주 (외부 의존) |
| 2 | `tauri.conf.json` Windows signing config | 0.5h |
| 3 | CI matrix (GitHub Actions) — Windows runner + signtool | 4h |
| 4 | `tauri-plugin-updater` 통합 + `latest.json` 생성 자동화 | 4h |
| 5 | Updater UI (modal "새 버전 0.x.x 사용 가능") | 2h |
| 6 | 첫 EV-signed release v1.1.0 출시 | 1h |
| 7 | SmartScreen 첫 인스톨 검증 (clean Windows VM) | 1h |

→ 외부 의존 제외 시 ~12.5h (1.5일).

### 위험 / 완화
- EV 인증서 발급 1~2주 (벤더 검증 절차) — 사전 신청 권장
- HSM 분실 / 손상 시 재발급 비용 — 백업 plan 필요
- SmartScreen 평판 즉시 통과 안 될 수 있음 → 사용자 README "More info → Run anyway" 가이드 유지

## 3. v1.2 — Sentry self-hosted + crash 수집 (~0.5개월)

### 진입 조건
- v1.1 출시 후 1주 — 사용자 본인 dogfood 안정
- VPS (Hetzner / DigitalOcean) ~$10/mo 예산

### 결정 항목 (4)
1. **self-host vs SaaS**:
   - SaaS Sentry $26/mo (Team) — 즉시 사용 가능
   - self-host Hetzner CX22 ~€5/mo + sentry-self-hosted Docker — 컨트롤
   - 권장: **self-host** — 회사 보안정책 친화 (plan §5 정합) + OPEX 절감
2. **데이터 정책**:
   - opt-in (default OFF) vs opt-out (default ON)
   - 권장: **opt-in 강제 default OFF** — 신뢰 우선, 사용자 명시 동의
3. **수집 범위**:
   - panic / IPC 에러 / 큰 graph 렌더 timeout
   - 권장: **panic + IPC 에러만** (UI/그래프는 사용자 실측 어려움)
4. **PII 마스킹**:
   - repo path / branch name 익명화 vs raw 전송
   - 권장: **path hash + branch 무전송** (CLAUDE.md 정합)

### 작업 분해
| # | 작업 | 시간 |
| - | --- | --- |
| 1 | Hetzner VPS 구성 + Docker compose Sentry self-hosted | 4h |
| 2 | DSN 발급 + Rust `sentry` crate 통합 (`tracing-subscriber` layer) | 3h |
| 3 | opt-in 토글 in Preferences > Privacy | 1h |
| 4 | PII 마스킹 (regex — path / branch) | 2h |
| 5 | 첫 panic 발생 시 dogfood 검증 | 1h |

→ ~11h (1.5일).

### 위험 / 완화
- self-host 의 운영 부담 (R5 burnout) — 매월 1회 업데이트만 (Docker compose pull)
- VPS 다운 시 사용자 영향? — opt-in 이라 무영향, retry queue 도 단순

- Apple Developer Program $99/yr 가입

### 결정 항목 (5)
1. **Apple Silicon vs Intel 우선순위**:
   - 권장: **universal binary** (`tauri build --target universal-apple-darwin`) — 1회로 양쪽 커버
2. **notarization**:
   - 권장: **자동화** (CI 에서 `notarytool submit`)
3. **테스터 채널**:
   - TestFlight (불가 — Tauri 는 App Store 외 배포)
   - GitHub releases + 직접 다운로드 (권장)
4. **macOS 한글 입력**:
   - NFC / NFD 차이 — `unicode_normalization` 적용 필수 (R2 정합)
   - 권장: macOS 회귀 테스트 스위트 추가 (한글 commit / branch / PR 메시지)
5. **메뉴바 / dock 통합**:

   - dock badge 알림 (Launchpad N PR)
   - 권장: **표준 메뉴 + dock badge** (macOS native 느낌)

## 5. v1.4 — Linux 베타 (~0.5~1개월)

### 진입 조건
- v1.3 macOS 출시 후 — Tauri cross-platform 검증된 상태

### 결정 항목 (4)
1. **배포 형식 우선순위**:
   - **AppImage** (모든 distro 호환, 권장 1순위)
   - **flatpak** (Fedora / GNOME 친화)
   - **deb / rpm** (Debian / Ubuntu / Fedora 네이티브)
   - **snap** (Ubuntu, Canonical 의존)
   - 권장: **AppImage 우선 → flatpak → deb·rpm** 순
2. **WebKitGTK 의존**:
   - WebKitGTK 4.1 (Tauri 2 요구)
   - 일부 distro (예: Ubuntu 22.04) 는 4.0 — 의존성 명시 필수
3. **한글 폰트 fallback**:
   - 일부 distro 에 한글 폰트 미설치 → 깨짐 가능
   - 권장: 앱 번들에 fallback 폰트 (NotoSansKR) 포함 또는 README 안내
4. **자동 업데이트**:
   - AppImage 의 자체 updater (`AppImageUpdate`) vs Tauri updater
   - 권장: **Tauri updater 통일** — Windows / macOS / Linux 동일 endpoint

### 작업 분해
| # | 작업 | 시간 |
| - | --- | --- |
| 1 | Ubuntu 22.04 / Fedora 39 dev VM 구성 | 2h |
| 2 | `tauri build` Linux target 빌드 (deb / AppImage) | 3h |
| 3 | flatpak manifest 작성 + flathub 검토 | 4h |
| 4 | WebKitGTK 의존성 명시 + dialog | 1h |
| 5 | 한글 폰트 fallback 검증 | 1h |
| 6 | dogfood Ubuntu / Fedora | 2h |
| 7 | 첫 v1.4.0 Linux 출시 | 1h |

→ ~14h (2일).

### 위험 / 완화
- WebKitGTK 4.0 ~ 4.1 호환성 문제 — Tauri Discord 모니터
- flathub 검토 1~3주 — 우선 AppImage 만 즉시 출시 후 flatpak deferred

## 6. v1.5 — OAuth (Gitea + GitHub) (~1개월)

### 진입 조건
- v1.3 macOS 베타 후 — cross-platform 안정
- 현재 PAT 방식 사용자 피드백 — OAuth 요청 시점

### 결정 항목 (4)
1. **redirect_uri 전략**:
   - localhost loopback (`http://127.0.0.1:PORT/callback`) — 사용자 환경 의존
   - Custom URL scheme (`git-fried://oauth/callback`) — Tauri deep link 활용 (C7 `3f19f19` 이미 있음)
   - 권장: **Custom URL scheme** — deep link 인프라 재사용
2. **GitHub OAuth App**:
   - public client_id / secret 노출 위험 → device flow vs PKCE
   - 권장: **device flow** (`https://github.com/login/device`) — secret 불필요
3. **Gitea OAuth**:
   - 인스턴스마다 admin 등록 필요 — 자동화 어려움
   - 권장: **PAT fallback 유지** + OAuth 옵션 추가 (admin 등록한 인스턴스만)
4. **token 저장**:
   - 기존 keyring 인프라 그대로 — 변경 0
   - refresh token 처리 추가

### 작업 분해
| # | 작업 | 시간 |
| - | --- | --- |
| 1 | GitHub device flow 구현 (`useOAuth` composable) | 4h |
| 2 | Gitea OAuth (Authorization Code + PKCE) — 인스턴스 admin 검증 후 | 6h |
| 3 | Custom URL scheme `git-fried://oauth/callback` 처리 | 2h |
| 4 | refresh token 자동 갱신 | 3h |
| 5 | UI — Forge setup 에 "OAuth 로 로그인" 버튼 추가 (PAT 옵션도 유지) | 2h |
| 6 | 회귀 — 한글 token name / 만료 처리 | 1h |

→ ~18h (2~3일).

### 위험 / 완화
- Gitea OAuth 인스턴스마다 admin 등록 → PAT fallback 유지로 회피
- GitHub OAuth App 등록 시 client_id 노출 → device flow 로 secret 불필요

## 7. v1.6 — 수익 모델 검토 (~1+ 개월, optional)

### 진입 조건
- v1.5 사용자 100+ — GitHub stars 또는 issue tracker 활동
- 사용자 본인 본업 시간 침해 가능성 시점

### 결정 항목 (5)
1. **라이선스 전환**:
   - 현재 MIT 유지 (단순, 기여자 친화)
   - Fair Source (GitButler 모델, 2년 후 MIT) — 유료화 진입 시
   - 권장: **MIT 유지 + Pro tier feature 분리**
2. **Pro tier 기능 후보**:
   - Cloud Workspace 같은 종속 — **거부 (plan §5)**
   - Team admin / SAML / audit log — 회사 단위 가치
   - Advanced AI (Claude Pro 자동 라우팅) — 사용자 BYOK 위반
   - 권장: **Team admin / 다중 라이센스 관리** 만 (기능 자체는 무료, 관리 도구 유료)
3. **가격**:
   - $5/mo (Solo Pro)
   - $15/user/mo (Team)
   - 권장: **Team $10/user/mo only** (Solo 무료 유지)
4. **결제 인프라**:
   - Stripe ~3% + $0.30 / 거래
   - Lemon Squeezy ~5% (한국 사업자 친화)
   - 권장: **Lemon Squeezy** — 부가세 자동 처리
5. **OSS 후원**:
   - GitHub Sponsors / Open Collective
   - 권장: **둘 다 활성화** + Pro tier 별도

### 작업 분해 (대략)
- License key 시스템 (~1주)
- Stripe / Lemon Squeezy 통합 (~1주)
- Team admin UI (~2주)
- 마케팅 페이지 (`git-fried.dev`) (~1주)

→ ~5주.

### 위험 / 완화
- 사용자 본업 충격 (R5 burnout) — Solo 무료 유지로 부담 작음
- 한국 사업자 등록 / 세금 — Lemon Squeezy 가 처리
