## Claude self-blind spot 패턴

1. `devMock + Playwright`로 “화면 대부분”을 본다고 가정하지만, 실제 위험은 Tauri shell, capability, native dialog, keyring, notification, file lock, WebView2 lifecycle에 있음.
2. “7 edge axes”를 충분조건처럼 다루지만, app zoom/system DPI, wheel burst, OS permission, reload/restore, large-repo perf 같은 desktop 축이 빠져 있음.
3. Phase 끝 Codex 2회만으로 충분하다고 가정하지만, 기존 c82 기록은 Codex가 중간에 Critical을 잡는 쪽이 ROI가 컸음을 보여줌.
4. god component LOC를 위험 proxy로 삼지만, 사용자 빈도와 데이터 손실 가능성이 더 높은 경로가 우선이어야 함.
5. “bug doc + fix commit”을 산출물로 보지만, regression spec/gate 없이는 audit 결과가 다음 sprint에서 다시 휘발될 가능성이 큼.

## 즉시 plan 수정 권고 top 3

1. §2.2를 `IPC command family × error class` 매트릭스로 바꾸고 worktree/search/tag/importer/hide/profile까지 명시한다.
2. §4를 phase-end 2회가 아니라 A-1/A-3/B-risk 종료 시점의 inline Codex checkpoint로 바꾼다.
3. §5 Done criteria에 “Critical/High는 regression test 또는 untestable 사유 + native gate 통과”를 추가한다.

## 보류 가능 항목

- multi-monitor, 외장 디스플레이, 잠금 화면 복귀는 Low 또는 ROI 불명확. 다만 system DPI/app zoom은 앱에 직접 zoom state가 있으므로 보류하지 말고 edge axis에 넣는 편이 낫다.
- tray/updater/single-instance는 현재 코드에서 핵심 플러그인 흔적이 약하므로, plan에는 `N/A 확인 row`로 두고 실제 도입 전까지 deep audit는 보류 가능.
- RTL은 기존 plan에서도 의도 배제 성격이 강하므로, “fallback 깨짐 없음” smoke만 유지하면 충분하다.
